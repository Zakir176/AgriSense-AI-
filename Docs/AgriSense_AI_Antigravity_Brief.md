---

## Hard Scope Rules — Read First

This is the single most important section. The original ZICTA proposal describes a much bigger system (live camera AI, real-time audio disease detection, predictive analytics, multi-farm scaling, subscription billing). **None of that is being built in Phase 1.** If you (Antigravity) are ever about to generate something that implies live camera streaming, trained-from-scratch ML models, real-time audio classification, or multi-tenant/billing logic — stop and flag it instead of building it. Default to the simplest implementation that satisfies the requirement below.

| Capability | Phase 1 reality |
|---|---|
| AI Visual Monitoring | Pretrained model (YOLOv8n or MediaPipe) run on **uploaded video files**, not a live stream |
| Audio disease detection | **NOT implemented as live inference.** A static library of pre-tagged sample clips, UI clearly labelled "research-stage / illustrative" |
| Smart alerts | Rule/threshold-based (e.g. >20% deviation from rolling average). **No ML.** |
| Analytics/recommendations | Basic charts + simple if/then rule-based recommendation text. **No predictive ML.** |
| Multi-farm support | Out of scope. Single farm (Prime Nest) only. |
| Billing/subscriptions | Out of scope entirely. |
| Authentication | One basic farm-operator login. No roles/permissions system needed. |

Any UI surface for AI Visual Monitoring or Audio must visibly label its Phase 1 status in the interface copy itself (not just in docs) — e.g. "Analyzing uploaded footage" rather than implying a live camera, and "Research preview — sample data" on the audio screen.

## Tech Stack

- **Frontend:** Vue 3 + Vite + Tailwind CSS
- **Backend:** FastAPI (Python 3.11+)
- **Database:** PostgreSQL
- **Charting:** Chart.js (or any lightweight Vue-compatible charting lib)
- **Deployment:** Frontend → Vercel, Backend + DB → Railway
- **AI Visual Monitoring:** Pretrained YOLOv8n (ultralytics) or MediaPipe, run server-side on uploaded clips

This mirrors the developer's existing deployment pattern (used previously on the Tajweed Recitation Coach project) — reuse known-good patterns, don't introduce new infra.

## Data Model

```
Farm
 - id, name, location

Batch
 - id, farm_id (FK), start_date, bird_count, breed, status (active|archived)

FeedWaterReading
 - id, batch_id (FK), date, feed_kg, water_litres, flagged_abnormal (bool)

GrowthSample
 - id, batch_id (FK), date, avg_weight_g, sample_size

MedicationEntry
 - id, batch_id (FK), date, medicine_type, dosage, outcome_note

Alert
 - id, batch_id (FK), type, message, severity, created_at, acknowledged (bool)

MediaClip
 - id, batch_id (FK), file_url, uploaded_at

InferenceResult
 - id, media_clip_id (FK), bird_count_est, movement_score, low_activity_windows (json)
```

## API Surface (FastAPI routers)

| Route prefix | Responsibility |
|---|---|
| `/farms`, `/batches` | CRUD for farm and batch entities |
| `/readings` | Create/list feed & water readings; compute 7-day rolling average; flag abnormal (>20% deviation default) |
| `/growth` | Create/list weight samples; compute growth rate per batch |
| `/medications` | Create/list medication entries + outcome notes |
| `/alerts` | List/acknowledge alerts (generated by rule logic, not a separate ML service) |
| `/inference/video` | Accept uploaded clip → run pretrained model → store + return `InferenceResult` |
| `/auth` | Simple login (single farm-operator account, no roles) |

**Architectural rule:** keep the Inference logic (`/inference/video`) in its own isolated module/service file, not tangled into the CRUD routers. This is intentional — if AI work runs out of time, the rest of the app must still work and demo cleanly without it.

## Frontend Views (Vue 3 SPA)

1. **Dashboard** — active alerts list + summary cards per batch (this is the primary judge-facing screen — make it look good)
2. **Batches** — create/list/archive batches
3. **Feed & Water** — entry form + time-series chart per batch
4. **Growth** — weight entry form + growth chart per batch
5. **Medications** — entry form + history table per batch
6. **AI Visual Monitor** — upload a video clip, trigger `/inference/video`, display results. **UI copy must say "uploaded footage" explicitly.**
7. **Audio Insight** — static page, pre-tagged sample clips, **UI copy must say "research preview"**
8. **Analytics** — trend charts + simple rule-based recommendation text
9. **Login** — single basic auth screen

Visual style: clean, minimal, professional — matches the developer's established preference (rejected gradient-heavy/flashy designs on prior projects in favor of restrained, dark-mode-friendly aesthetics). Avoid generic AI-template look.

## Definition of Done for Phase 1 Demo

The app is demo-ready when:
- A user can create a batch, log feed/water/growth/medication data, and see it reflected on the dashboard
- At least one abnormal reading produces a visible alert
- A real video clip can be uploaded and produces a genuine (not hardcoded) inference result
- Every AI-related screen honestly labels its Phase 1 status in the UI itself
- The whole flow works end-to-end on the deployed Vercel/Railway instance, not just locally

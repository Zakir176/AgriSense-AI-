1. Introduction
1.1 Purpose
This Software Requirements Specification (SRS) defines the functional and non-functional requirements for AgriSense AI Phase 1 — the prototype version of an intelligent agricultural monitoring platform being developed for entry into the 2026 ZICTA ICT Innovation Programme. It is intended to give a shared, accurate reference for what will be built, by whom, and by when, so that all stakeholders evaluate the same scope.
1.2 Scope
AgriSense AI is a multi-modal monitoring platform for poultry farms, piloted at Prime Nest Poultry Farm. The long-term vision (as described in the ZICTA proposal) includes computer-vision-based behavior tracking, audio-based disease detection, feed/water tracking, growth monitoring, medication logging, and predictive analytics.
This document specifies the Phase 1 prototype only: the subset of that vision that is achievable by a single developer, alongside full-time academic commitments, within the build window ending at the ZICTA Final Pitching Event. Section 1.4 defines exactly what is in and out of scope for this phase, and this distinction is the most important part of the document.

2. Overall Description
2.1 Product Perspective
AgriSense AI Phase 1 is a standalone web application: a Vue 3 single-page frontend backed by a FastAPI service and a PostgreSQL database. It is a new system with no existing infrastructure to integrate with. It is designed so that the Phase 1 rule-based and demo-scoped components can later be replaced by genuine production ML services without a frontend or data-model rewrite (see Architecture Document, Section 6: Future Evolution Path).
2.2 Product Functions (Summary)
Record and view daily feed and water consumption per batch
Record weekly sampled chicken weights and view growth trends per batch
Log medication and treatments administered, linked to batches
Generate threshold-based alerts (e.g. feed consumption drop, abnormal water intake)
Run AI-based visual inference on uploaded/recorded farm video to flag inactivity or abnormal movement
Present a basic analytics view with trend charts and rule-based recommendations
Present an early-stage / illustrative view of audio-based distress detection, clearly labelled as a research-stage feature

2.4 Operating Environment
Frontend: modern browser (Chrome, Edge, Safari) on desktop or mobile
Backend: FastAPI (Python 3.11+), deployed on Railway or similar PaaS
Database: PostgreSQL
Hosting: Vercel (frontend) and Railway (backend), consistent with the developer's existing deployment experience
2.5 Design and Implementation Constraints
Single developer, part-time around a full-time academic course load with weekly assessments
No dedicated hardware (camera/microphone) procured for Phase 1 — video/audio inputs are pre-recorded samples supplied by the farm, not a live IoT feed
Hard external deadline: ZICTA Final Pitching Event (see Project Schedule, Section 2.7)
No budget for paid ML training infrastructure — Phase 1 AI components use free-tier or open pretrained models only
2.6 Assumptions and Dependencies
Evans/the farm will supply at least one short video clip of chickens and baseline feed/water figures for realistic demo data (see open dependency tracker, Appendix A)
ZICTA workshop dates and the pitching event date do not shift from the published 2026 cohort schedule
Internet connectivity is available at the farm and pitching venue for any live-hosted components
2.7 Project Schedule Context
Phase 1 development runs from late June 2026 to the ZICTA Final Pitching Event (7–11 August 2026). This window includes the developer's ongoing weekly academic assessments and two further ZICTA programme workshops (Technical Development Workshop 1, 14–16 July; Business Development Workshop, 27–31 July). University exams begin 23 August 2026, after the pitching event, and are therefore a post-pitch constraint rather than a build-time constraint. A detailed week-by-week build schedule is maintained separately and updated as the programme progresses.

4. Non-Functional Requirements
4.1 Performance
Dashboard pages shall load within 3 seconds on a standard broadband or 4G connection.
Video inference (FR-6.2) may run asynchronously; result shall be available within a reasonable processing window communicated to the user (e.g. “Analyzing — results in ~30s”), not necessarily real-time.
4.2 Usability
The UI shall be usable by a non-technical farm operator without training, consistent with the existing established Vue/Tailwind design conventions used in the developer's prior projects.
All AI/demo-scoped features (Sections 3.6, 3.7) shall carry a visible UI label distinguishing them from fully production-ready features, per the disclosure principle in Section 1.4.
4.3 Reliability
Core CRUD features (Sections 3.1–3.5) shall not depend on any external AI service being available; the dashboard must function fully if the AI modules are offline.
4.4 Security
Basic authentication shall gate access to the dashboard (single farm-operator account is sufficient for Phase 1).
All data in transit shall use HTTPS.
4.5 Maintainability
Code shall be organized to allow Phase 2 replacement of rule-based logic with trained models without restructuring the data model (see Architecture Document).
4.6 Portability
The system shall run on any standard modern browser without browser-specific code.

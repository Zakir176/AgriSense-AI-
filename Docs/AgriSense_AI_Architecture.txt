1. Architectural Overview
1.1 Goals
Deliver a credible, demoable Phase 1 prototype within a 7-week solo-developer window
Keep the core data model and API stable so Phase 2 can swap in real ML services without breaking the frontend
Reuse the developer's existing, proven stack (Vue 3 + Python/FastAPI) rather than introducing new unfamiliar tools under time pressure
Make the boundary between вҖңrealвҖқ and вҖңdemo-scopedвҖқ components explicit at the architecture level, not just in product copy
1.2 Architectural Style
AgriSense AI Phase 1 uses a classic three-tier web architecture: a Vue 3 single-page application (presentation tier), a FastAPI REST service (application/logic tier), and PostgreSQL (data tier). AI/ML components are isolated behind a dedicated internal module boundary (the вҖңInference ServiceвҖқ) so they can be swapped, scaled out to their own process, or replaced entirely in Phase 2 without touching the core CRUD logic.

2.3 Inference Service (AI Boundary Module)
This module exists specifically to keep Phase 1's вҖңdemo-scopedвҖқ AI honest and isolated. It exposes one simple internal interface вҖ” given a video file, return a structured result (e.g. bird count estimate, movement score, flagged low-activity windows) вҖ” regardless of what's running underneath.
Phase 1 implementation: a pretrained, off-the-shelf object detection/tracking model (e.g. YOLOv8n or MediaPipe) run against the uploaded clip вҖ” genuine inference, not hardcoded output, but not custom-trained on Prime Nest data
Audio module: NOT a live inference path in Phase 1 вҖ” served as static pre-tagged sample data from the database, with UI copy that says so
Phase 2 swap-in point: this module's interface is designed so a custom-trained model or live camera/audio pipeline can replace the implementation without changing any caller code
2.4 Data Tier (PostgreSQL)
A single relational schema serves all modules. Schema is intentionally simple for Phase 1 вҖ” normalized enough to avoid rework, not over-engineered for scale that isn't needed yet.

4. Key System Flows
4.1 Daily Feed/Water Entry вҶ’ Alert Flow
1. User submits feed/water reading via frontend form2. POST /readings stores the reading3. Rules engine computes 7-day rolling average for the batch4. If deviation > threshold (default 20%):     вҶ’ reading flagged abnormal     вҶ’ Alert record created5. Dashboard polls/fetches /alerts and surfaces it to the user
4.2 AI Visual Monitoring Flow
1. User uploads a short video clip via the AI Visual Monitor view2. Frontend POSTs file to /inference/video3. Backend stores clip as MediaClip, passes file to Inference Service4. Inference Service runs pretrained detection/tracking model5. Structured result (bird count est., movement score, flagged windows)   stored as InferenceResult and returned to frontend6. Frontend renders result with explicit вҖңuploaded footage, Phase 1вҖқ label
